## DHCP-friendly K3s clustering with Tailscale

The original [chapter **G025**](G025%20-%20K3s%20cluster%20setup%2008%20~%20K3s%20Kubernetes%20cluster%20setup.md) builds the K3s cluster assuming you can assign **static IPs to the primary NICs (ens18)** from your router. But many environments — ISP-provided routers with locked-down DHCP, CGNAT, or networks managed by someone else — do not let you reserve static addresses. Without a stable IP on ens18, the `node-external-ip` changes on every lease renewal, and remote `kubectl` clients lose their connection.

This chapter adds **[Tailscale](https://tailscale.com/)** — a WireGuard-based mesh VPN — to give each node a stable identity that survives DHCP churn. Critically, **Tailscale does not replace the existing cluster networking.** The isolated bridge (`vmbr1`) and the secondary NICs (`ens19`) remain exactly as described in G025. Tailscale addresses only the one point of fragility: the ens18 static IP dependency.

> [!IMPORTANT]
> **This chapter is a modification of G025 — not a complete rewrite**
> Sections that are identical to G025 (the folder structure for K3s config files, graceful shutdown setup, `kubelet.conf`, the cleanup pods service, enabling kubectl autocompletion, log rotation, VM startup/shutdown ordering) are **not repeated**. When you reach such a step, follow the corresponding section in [chapter **G025**](G025%20-%20K3s%20cluster%20setup%2008%20~%20K3s%20Kubernetes%20cluster%20setup.md). This chapter only documents what **changes**.

## Why Tailscale — and why not replace everything

### What already works without the router

Look at the two network cards in the G025 design:

| NIC | IP source | Needs router cooperation? |
|---|---|---|
| **ens18** (external/LAN) | Static lease from router | ✅ YES — this is the problem |
| **ens19** (internal cluster) | Locally set in `/etc/network/interfaces` | ❌ NO — purely inside Proxmox VE |

The `172.16.x.x` IPs on ens19 are configured **inside each VM's filesystem** and communicate through the isolated `vmbr1` bridge. The router never sees these addresses and has no say over them. They are already stable regardless of what the router does.

This means **flannel, the K3s API server, kubelet, CoreDNS, and all pod-to-pod traffic already work without router cooperation.** Tearing out ens19 to replace it with Tailscale would trade a perfectly stable local bridge for a WireGuard tunnel that adds encryption latency between pods on the same physical host and introduces a dependency on Tailscale's coordination server.

### What Tailscale fixes: the ens18 identity problem

The only thing that breaks without router static IPs is the **ens18 address**. This address serves two roles in G025:

1.  **`node-external-ip`** — the IP that Kubernetes reports as the node's external identity.
2.  **`kubectl` target** — remote clients connect to `https://10.4.1.1:6443`.

Tailscale gives every VM a **permanent `100.x.y.z` address** that survives DHCP changes, router reboots, and even physical network moves. Use that address for `node-external-ip` and for kubectl connections, and the cluster becomes immune to DHCP churn — without touching the ens19 internal networking at all.

## The hybrid architecture

### Side-by-side comparison

```
       G025 (original)                        G025.1 (hybrid — this chapter)

ens18: 10.4.1.1 (static, router)       ens18: DHCP (any — router doesn't matter)
ens19: 172.16.1.1 (local, static)      ens19: 172.16.1.1 (local, static — unchanged)
                                       tailscale0: 100.x.y.z (stable, WireGuard)

flannel-iface:     ens19               flannel-iface:     ens19        ← same
advertise-address: 172.16.1.1          advertise-address: 172.16.1.1   ← same
node-ip:           172.16.1.1          node-ip:           172.16.1.1   ← same
node-external-ip:  10.4.1.1            node-external-ip:  100.x.y.z    ← changed
tls-san IP:        10.4.1.1            tls-san IP:        100.x.y.z    ← changed
kubectl connects:  https://10.4.1.1    kubectl connects:  https://100.x.y.z ← changed

Agent server URL:  https://172.16.1.1  Agent server URL:  https://172.16.1.1  ← same
```

**What stays identical to G025:**
- Two NICs per VM (ens18 + ens19).
- The isolated `vmbr1` bridge on the Proxmox host.
- Flannel runs on ens19 with `host-gw` backend — fast, local, no encryption overhead.
- Agents join the server via the ens19 IP (`172.16.1.1:6443`).
- All inter-node cluster traffic flows through the local bridge.

**What changes:**
- ens18 uses DHCP instead of a router-assigned static IP.
- Tailscale runs on each VM, providing a stable `100.x.y.z` address.
- `node-external-ip` points to the Tailscale address.
- Remote `kubectl` clients connect through the Tailscale mesh, not the LAN IP.
- The PVE firewall no longer needs to open port 6443 on ens18 for kubectl.

## Prerequisites

### Tailscale account and tags

1.  Create a free [Tailscale account](https://login.tailscale.com/start) if you do not have one.
2.  In the [Tailscale admin console](https://login.tailscale.com/admin), go to **Settings → ACLs** (or **Access Controls**).
3.  Define a tag for your K3s cluster nodes so they do not consume user device slots:

    ~~~json
    "tagOwners": {
      "tag:k3s-node": ["autogroup:admin"]
    }
    ~~~

### Reusable pre-authentication key

Generate an auth key so each VM can join without an interactive browser login:

1.  In the Tailscale admin console, go to **Settings → Keys**.
2.  Click **Generate auth key**.
3.  Configure:
    - **Description**: `K3s cluster nodes`
    - **Reusable**: ✅ Enabled
    - **Ephemeral**: Leave disabled
    - **Tags**: Select `tag:k3s-node`
4.  Click **Generate key** and **copy it immediately**. It starts with `tskey-auth-`.

> [!WARNING]
> **Protect this auth key**
> Anyone with this key can add a device to your Tailscale network tagged as `k3s-node`. Store it in a password manager.

## Adapting the VM template for Tailscale

Start from the `k3snodetpl` template created in [chapter **G024**](G024%20-%20K3s%20cluster%20setup%2007%20~%20K3s%20node%20VM%20template%20setup.md). This template already has ens19 configured with a static `172.31.254.x` address and connected to `vmbr1` — you will keep that.

### Full clone from k3snodetpl

1.  In the Proxmox VE web console, go to `k3snodetpl` → `More` → `Clone`.
2.  Make a **Full Clone** so you can freely modify the filesystem:

    | Field | Value |
    |---|---|
    | **Mode** | `Full Clone` |
    | **VM ID** | `103` (or any available ID) |
    | **Name** | `k3stailtpl.homelab.cloud` |
    | **Target Storage** | `ssd_disks` |

3.  Click `Clone`.

### Switch ens18 to DHCP

The template's ens18 was configured for DHCP in the original G024 — leave it as DHCP. If for some reason it was changed, verify:

~~~sh
$ cat /etc/network/interfaces
~~~

The ens18 block should read:

~~~sh
# The primary network interface
allow-hotplug ens18
iface ens18 inet dhcp
~~~

> [!NOTE]
> **ens19 stays exactly as G024 configured it**
> Do not remove or change the ens19 block. Its static `172.31.254.x` address is set locally and works without the router. Only the next step — changing that IP to match a specific K3s node — differs from the template default, and that happens after cloning, just as in G025.

### Install Tailscale inside the VM

1.  Add the Tailscale repository and install:

    ~~~sh
    $ curl -fsSL https://pkgs.tailscale.com/stable/debian/trixie.noarmor.gpg | \
      sudo tee /usr/share/keyrings/tailscale-archive-keyring.gpg > /dev/null
    $ curl -fsSL https://pkgs.tailscale.com/stable/debian/trixie.tailscale-keyring.list | \
      sudo tee /etc/apt/sources.list.d/tailscale.list
    $ sudo apt update
    $ sudo apt install -y tailscale
    ~~~

    > [!NOTE]
    > **Replace `trixie` with your VM's Debian codename**
    > Run `lsb_release -cs` to confirm.

2.  Test that Tailscale works by joining with your auth key:

    ~~~sh
    $ sudo tailscale up --authkey tskey-auth-kXXXXXXXXXX --accept-dns=false --accept-routes=false
    $ tailscale status
    100.64.0.1    k3stailtpl          tagged-devices   linux   active; direct ...
    ~~~

3.  Log out, then **purge the Tailscale state files** so cloned VMs do not share the same machine identity:

    ~~~sh
    $ sudo tailscale logout
    $ sudo rm -rf /var/lib/tailscale/
    ~~~

    > [!WARNING]
    > **This step is critical for clones**
    > `tailscale logout` unregisters the device from your Tailscale network, but the machine key and node key files under `/var/lib/tailscale/` survive on disk. If you clone the VM without deleting this directory, every clone presents the same identity to Tailscale and you get the error: *"Multiple cloned devices are trying to share a key. Reinstall."* Purging `/var/lib/tailscale/` forces each clone to generate fresh keys on its first `tailscale up`.

    After the purge, the `/var/lib/tailscale/` directory is gone and the `tailscale0` interface has disappeared — the template is clean for cloning.

### Convert to a Tailscale-aware VM template

1.  Shut down the VM: `sudo shutdown now`
2.  In Proxmox VE, select the VM → `More` → `Convert to template`.
3.  [Make a manual backup of this template](G023%20-%20K3s%20cluster%20setup%2006%20~%20Debian%20VM%20template%20and%20backup.md#vm-templates-backup), then restore it to reclaim space from empty blocks.

Your `k3stailtpl` template is ready. It has:
- **ens18** — DHCP, LAN-facing NIC.
- **ens19** — static `172.31.254.1` configured locally, connected to `vmbr1` (will be changed per-node).
- **Tailscale** — installed but with **no state files** (`/var/lib/tailscale/` is empty). Each clone will generate a fresh identity on its first `tailscale up`.
- **All G024 hardening** — sysctl parameters, disabled swap, renamed VG, SSH/Fail2Ban/TOTP — intact.

> [!NOTE]
> **If you already cloned the template before purging state**
> On each existing clone, run `sudo rm -rf /var/lib/tailscale/` before `tailscale up`. No need to rebuild the VMs.

## Creating the K3s cluster nodes

> [!WARNING]
> **Before booting any clone: ensure Tailscale state was purged on the template**
> If you cloned `k3stailtpl` before the `/var/lib/tailscale/` purge step was added to this chapter, each clone shares the same machine key. Tailscale will reject them with *"Multiple cloned devices are trying to share a key."* 
>
> **Fix**: on each clone that hasn't yet joined Tailscale, boot once and run `sudo rm -rf /var/lib/tailscale/`. No need to rebuild the VMs.

### Clone the VMs

From `k3stailtpl`, create **linked clones** for each K3s node:

| VM | VM ID | Name |
|---|---|---|
| Server | `411` | `k3sserver01.homelab.cloud` |
| Agent 1 | `421` | `k3sagent01.homelab.cloud` |
| Agent 2 | `422` | `k3sagent02.homelab.cloud` |

> [!NOTE]
> **VM IDs no longer encode the ens18 IP**
> In G025, VM ID `411` was chosen because ens18 was `10.4.1.1`. With DHCP on ens18, the correlation no longer matters. The IDs above are suggestions — use whatever you prefer.

Adjust Memory and Processors for each VM as described in [G025's resource sections](G025%20-%20K3s%20cluster%20setup%2008%20~%20K3s%20Kubernetes%20cluster%20setup.md#vm-as-k3s-server-node).

### Preparing each VM

Boot each VM and SSH in as `mgrsys`. Since ens18 uses DHCP, use the PVE console (open the VM's `Console` tab in Proxmox VE) or your router's device list to find its current address.

> [!WARNING]
> **The order of these steps matters**
> Tailscale registers the VM's hostname as its device name on `tailscale up`. If you join Tailscale before changing the hostname, the device will be registered with the template's name and must be manually renamed from the Tailscale admin console. Follow the order below.

**1. Hostname** — Change this **first**, before any networking or Tailscale step. For each VM, use `hostnamectl set-hostname` and edit `/etc/hosts` as described in [G025's hostname section](G025%20-%20K3s%20cluster%20setup%2008%20~%20K3s%20Kubernetes%20cluster%20setup.md#customizing-the-hostname). Log out and back in to see the new hostname in your prompt.

| VM | Hostname |
|---|---|
| `k3sserver01` | `k3sserver01` |
| `k3sagent01` | `k3sagent01` |
| `k3sagent02` | `k3sagent02` |

**2. ens19 static IP** — Same as G025, but no router involvement. Edit `/etc/network/interfaces` and change only the `address` line in the ens19 block:

| VM | ens19 address |
|---|---|
| `k3sserver01` | `172.16.1.1` |
| `k3sagent01` | `172.16.2.1` |
| `k3sagent02` | `172.16.2.2` |

Apply: `sudo ifdown ens19; sudo ifup ens19`

**3. Tailscale** — Now that the hostname is correct, authenticate with the reusable key:

~~~sh
$ sudo tailscale up --authkey tskey-auth-kXXXXXXXXXX \
    --accept-dns=false --accept-routes=false \
    --hostname=$(hostname)
~~~

Using `$(hostname)` picks up the hostname you set in step 1 — no risk of a copy-paste mistake.

**4. TOTP, SSH keys, password** — Change these per-node as described in [G025's credential sections](G025%20-%20K3s%20cluster%20setup%2008%20~%20K3s%20Kubernetes%20cluster%20setup.md#changing-the-totp-code).

### Record each node's addresses

Run the following on each VM and write down the results:

~~~sh
$ tailscale ip
100.73.42.101
~~~

Example assignment (your actual Tailscale IPs will differ):

| VM | ens19 IP | Tailscale IP |
|---|---|---|
| `k3sserver01` | `172.16.1.1` | `100.73.42.101` |
| `k3sagent01` | `172.16.2.1` | `100.73.42.102` |
| `k3sagent02` | `172.16.2.2` | `100.73.42.103` |

> [!IMPORTANT]
> **Write down the Tailscale IPs.**
> You need the server's Tailscale IP for `node-external-ip`, `tls-san`, and the kubectl `kubeconfig`.

## Firewall

### What changes

In G025, the PVE firewall on the ens18 NIC opened ports `22`, `80`, `443`, `6443` — including the K3s API port for LAN-based kubectl clients. In this hybrid design, kubectl clients connect through Tailscale, so **port 6443 on ens18 is no longer needed.**

### PVE rules for the DHCP NIC (ens18)

Create a security group `k3s_tailscale_net0_in` with:

- **Rule 1**: Type `in`, Action `ACCEPT`, Protocol `tcp`, Source `local_network_ips`, Dest. port `22`, Comment `SSH for entire local network`.

For agent nodes, also add:

- **Rule 2**: Type `in`, Action `ACCEPT`, Protocol `tcp`, Source `local_network_ips`, Dest. port `80`, Comment `HTTP`.
- **Rule 3**: Type `in`, Action `ACCEPT`, Protocol `tcp`, Source `local_network_ips`, Dest. port `443`, Comment `HTTPS`.

Apply this security group to each VM's firewall on the `net0` interface. Also enable `IP filter` with an IP set containing the DHCP-assigned IP (or the LAN subnet).

> [!NOTE]
> **No 6443 rule on ens18**
> kubectl clients connect through the Tailscale mesh. The Traffic never touches the PVE bridge, so no firewall rule is needed. If you ever need LAN-based kubectl access as a fallback, add the 6443 rule back.

### ens19 and the isolated bridge — no change

The ens19 NICs are not firewalled in the PVE firewall — they sit on the isolated `vmbr1` bridge. This is identical to G025 and requires no modification.

## K3s Server node setup

The folder structure, `kubelet.conf`, and cleanup service are identical to [G025's server setup section](G025%20-%20K3s%20cluster%20setup%2008%20~%20K3s%20Kubernetes%20cluster%20setup.md#k3s-server-node-setup). Create those files exactly as described there. Only `config.yaml` changes.

### The k3sserver01 config.yaml for the hybrid design

Create `/etc/rancher/config.yaml.d/config.yaml` (symlinked from `/etc/rancher/k3s/config.yaml`) with:

~~~yaml
# k3sserver01 — Tailscale hybrid

cluster-domain: "homelab.cluster"
tls-san:
  - "k3sserver01.homelab.cloud"
  - "100.73.42.101"
flannel-backend: host-gw
flannel-iface: "ens19"
bind-address: "0.0.0.0"
https-listen-port: 6443
advertise-address: "172.16.1.1"
advertise-port: 6443
node-ip: "172.16.1.1"
node-external-ip: "100.73.42.101"
node-taint:
  - "node-role.kubernetes.io/control-plane=true:NoSchedule"
kubelet-arg: "config=/etc/rancher/k3s/kubelet.conf"
disable:
  - metrics-server
  - servicelb
protect-kernel-defaults: true
secrets-encryption: true
agent-token: "SomeReallyLongPassword"
~~~

**What changed from G025 — and what didn't:**

| Parameter | G025 value | G025.1 value | Reason |
|---|---|---|---|
| `flannel-iface` | `"ens19"` | `"ens19"` | **Unchanged** — flannel still uses the local isolated bridge |
| `advertise-address` | `"172.16.1.1"` | `"172.16.1.1"` | **Unchanged** — agents still join via the local bridge |
| `node-ip` | `"172.16.1.1"` | `"172.16.1.1"` | **Unchanged** — node identity on the internal network |
| `node-external-ip` | `"10.4.1.1"` | `"100.73.42.101"` | **Changed** — Tailscale IP replaces the router-dependent ens18 address |
| `tls-san` IP | `"10.4.1.1"` | `"100.73.42.101"` | **Changed** — kubectl clients connect via Tailscale, so the SAN must match |

> [!IMPORTANT]
> **Replace `100.73.42.101` with YOUR server's actual Tailscale IP**
> Run `tailscale ip` on the server VM and use that value in both `node-external-ip` and `tls-san`.

### Install the K3s server

Identical to G025:

~~~sh
$ wget -qO - https://get.k3s.io | INSTALL_K3S_VERSION="v1.33.4+k3s1" sh -s - server
~~~

Verify:

~~~sh
$ sudo kubectl get nodes -o wide
NAME          STATUS   ROLES                  AGE   VERSION        INTERNAL-IP   EXTERNAL-IP      ...
k3sserver01   Ready    control-plane,master   30s   v1.33.4+k3s1   172.16.1.1    100.73.42.101    ...
~~~

`INTERNAL-IP` is the ens19 address. `EXTERNAL-IP` is the Tailscale address. The DHCP address on ens18 is invisible to Kubernetes — exactly as intended.

## Set up kubectl remote access through Tailscale

In G025, you opened port 6443 on the PVE firewall and connected via the server's ens18 static IP. With the hybrid design, you connect through Tailscale — no firewall rule needed.

### Install Tailscale on your client machine

Install Tailscale on your laptop or desktop from [tailscale.com/download](https://tailscale.com/download). Run `tailscale up` (interactive login on a personal device — no auth key needed). Verify connectivity:

~~~sh
$ tailscale status | grep k3sserver01
100.73.42.101  k3sserver01        tag:k3s-node   linux   active; direct ...
$ ping 100.73.42.101
~~~

### Get and adapt the kubeconfig

1.  From the server node, copy `/etc/rancher/k3s/k3s.yaml` to your client.
2.  Place it at `$HOME/.kube/config`:

    ~~~sh
    $ mkdir -p $HOME/.kube
    $ mv k3s.yaml $HOME/.kube/config
    $ chmod 600 $HOME/.kube/config
    ~~~

3.  Edit `$HOME/.kube/config` and change the `server:` line to the **server's Tailscale IP**:

    ~~~yaml
    server: https://100.73.42.101:6443
    ~~~

    > [!NOTE]
    > **You do NOT need to open port 6443 on the PVE firewall**
    > The connection travels through Tailscale's WireGuard tunnel. The PVE firewall never sees this traffic.

### Connect and verify

~~~sh
$ kubectl cluster-info
Kubernetes control plane is running at https://100.73.42.101:6443
CoreDNS is running at https://100.73.42.101:6443/api/v1/namespaces/kube-system/services/kube-dns:dns/proxy
~~~

You can now manage the cluster from anywhere your Tailscale client has connectivity — not just the LAN. The TLS handshake succeeds because `100.73.42.101` is listed in the server's `tls-san`.

> [!TIP]
> **Fallback: LAN-based kubectl**
> If you ever need LAN-based kubectl access (e.g., Tailscale is down), open port 6443 on the PVE firewall for ens18, determine the server's current DHCP address (with `ip a show ens18` on the server), and point kubeconfig at that IP. This works because `tls-san` includes the hostname `k3sserver01.homelab.cloud`, which you can resolve locally.

## Considerations for deployed services

### MetalLB

[MetalLB](G027%20-%20K3s%20cluster%20setup%2010%20~%20Deploying%20the%20MetalLB%20load%20balancer.md) in L2 mode announces IPs via ARP on a specified interface. Announce on `ens18` with an IP range from your LAN subnet. This works even with DHCP because MetalLB uses its own VIPs — it does not need the host's IP to be static. The only requirement is that the MetalLB IPs do not overlap with your router's DHCP pool.

For LoadBalancer services accessible through Tailscale, you can also use [Tailscale Serve](https://tailscale.com/kb/1242/tailscale-serve) or [Tailscale Funnel](https://tailscale.com/kb/1223/tailscale-funnel) to expose individual services without MetalLB.

### cert-manager and ingress

[Traefik](G030%20-%20K3s%20cluster%20setup%2013%20~%20Enabling%20the%20Traefik%20dashboard.md) binds to all interfaces by default, so it listens on ens18, ens19, and tailscale0. The [cert-manager setup](G029%20-%20K3s%20cluster%20setup%2012%20~%20Setting%20up%20cert-manager%20and%20self-signed%20CA.md) and ingress configurations from the main guide work unchanged.

If you want to access services via Tailscale hostnames, add the Tailscale MagicDNS name (e.g., `k3sserver01.your-tailnet.ts.net`) to your IngressRoutes and TLS certificates.

### NUT client

The [NUT client setup in G022](G022%20-%20K3s%20cluster%20setup%2005%20~%20Connecting%20the%20VM%20to%20the%20NUT%20server.md) connects to the Proxmox VE host's NUT server. The VMs reach the PVE host through ens18 on the `vmbr0` bridge — the same LAN they have always been on. If the PVE host's IP is stable (it typically is — it is the Proxmox host itself), NUT works exactly as in G022. If even the PVE host's IP is not stable, run Tailscale on the PVE host and point NUT clients to the host's Tailscale IP.

## Relevant system paths

### On each K3s node VM

- `/etc/rancher/config.yaml.d/config.yaml`
- `/etc/rancher/config.yaml.d/kubelet.conf`
- `/etc/rancher/k3s/config.yaml` → symlink to above
- `/etc/rancher/k3s/kubelet.conf` → symlink to above
- `/etc/network/interfaces`
- `/etc/apt/sources.list.d/tailscale.list`
- `/var/lib/rancher/k3s/server/token` (server node only)

### On the kubectl client machine

- `$HOME/.kube/config`
